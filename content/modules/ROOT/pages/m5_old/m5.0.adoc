== Lab 5 - Protect LLM consumption with Connectivity Link



In the previous modules, you unified chat platforms, ingested client data, and introduced an AI agent that resolves invoice issues through natural language -- all powered by _Apache Camel_, _Kafka_, and LLM APIs.



But giving teams access to Large Language Models introduces a new challenge: **cost control**. AI costs are driven by token consumption, not simple API hits. Traditional rate limiting -- which just counts requests -- can't tell the difference between a 10-token "say hello" prompt and a 10,000-token "summarize this document" prompt. Both count as one request, but one costs 100x more than the other. Without token-level visibility, one runaway prompt -- or one team with no guardrails -- can burn through your AI budget unchecked.

In this lab, you'll place a **governance layer** in front of the LLM using _Red Hat Connectivity Link_ and implement two policies:

* **AuthPolicy** -- Control _who_ can access the LLM API using API key authentication and group-based authorization.
* **TokenRateLimitPolicy** -- Control _how much_ each user can consume by enforcing rate limits based on actual token usage.

By the end, the AI agent will have tiered access control -- _free_ users get a limited token budget, while _gold_ users get a higher allowance.

=== Why Connectivity Link?

_Red Hat Connectivity Link_ is built on the link:https://kuadrant.io/[_Kuadrant_ project] and the _Kubernetes Gateway API_ standard. What makes it different:

* **Token-aware rate limiting** -- purpose-built for AI workloads. Limits are based on actual tokens consumed, not request counts.
* **Zero code changes** -- policies attach to _Gateway API_ resources (Gateways, HTTPRoutes). Your AI agent and application code stay untouched.
* **Kubernetes-native** -- policies are standard CRDs, applied with `oc apply`, stored in Git, reviewed in PRs. No separate management console or vendor control plane.
* **Composable** -- AuthPolicy extracts identity, TokenRateLimitPolicy uses it for per-user budgets. Policies layer on top of each other declaratively.

{empty} +

=== The Gateway API Model

The _Kubernetes Gateway API_ is the standard for managing traffic routing in _Kubernetes_ -- a more flexible, layered replacement for the older Ingress API. _Connectivity Link_ builds on this model, which is what makes all of the above possible. Here's how the pieces fit together:

* **GatewayClass + Gateway** -- owned by the platform team. Defines _how_ traffic enters the cluster. Already provisioned for you.
* **HTTPRoute** -- owned by development team. Defines _where_ your traffic goes (in this case, the LLM backend).
* **Policies** -- _Connectivity Link_ attaches AuthPolicy and TokenRateLimitPolicy to these resources. Governance is enforced at the infrastructure level -- no changes to your application code.

This layered model means you can add authentication and token budgets to any service behind an HTTPRoute without touching the service itself.
