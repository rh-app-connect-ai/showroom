== Lab 5 - Protect LLM consumption with Connectivity Link

== Introduction

In the previous phases of this workshop, you connected disparate chat platforms into a unified communication hub, securely ingested and transformed client data, and introduced an AI agent that helps support agents resolve invoice issues through natural language -- all powered by _Apache Camel_, _Kafka_, and LLMs.

But providing internal teams with access to Large Language Models introduces new challenges. AI resource costs are tied to **token consumption** rather than simple API hits. Without proper governance, there is no way to prevent system abuse or control runaway expenses. Traditional rate limiting -- which simply counts requests -- is not enough.

In this final phase, you will place a **governance layer** in front of the LLM using _Red Hat Connectivity Link_. You will implement two key policies:

* **AuthPolicy** -- Control _who_ can access the LLM API using API key-based authentication and group-based authorization.
* **TokenRateLimitPolicy** -- Control _how much_ each user can consume by enforcing rate limits based on actual LLM token usage.

By the end of this module, the AI agent will be fully secured with tiered access control -- _free_ users get a limited token budget, while _gold_ users enjoy a higher allowance.

== Introducing Red Hat Connectivity Link

=== What is Connectivity Link?

We will be Red Hat Connectivity Link across this module to implement the governance policies for the LLM. _Red Hat Connectivity Link_ is a _Kubernetes_-native solution for managing, securing, and controlling API traffic. Built on the link:https://kuadrant.io/[_Kuadrant_ project], it integrates directly with the _Gateway API_ to enforce policies at the gateway level and HTTP Route Level -- before requests ever reach the LLM backend.

{empty} +

=== Gateway API Architecture

Gateway API is an official Kubernetes project focused on L4 and L7 routing in Kubernetes. This project represents the next generation of Kubernetes Ingress, Load Balancing, and Service Mesh APIs. It uses a layered model:

[.mermaid]
[subs="attributes"]
----
flowchart LR
    subgraph yoursLabel [" "]
        subgraph yours ["OpenShift"]
            GC[GatewayClass]
            GT[Gateway]
            HTTP[HTTPRoute]
        end
    end

    GC --> GT --> HTTP

    style yoursLabel fill:none,stroke:none
    style GC fill:#fff,stroke:#7986cb
    style GT fill:#fff,stroke:#7986cb
    style HTTP fill:#fff,stroke:#7986cb
----

[.mermaid-hints-simple]
****
GC | GatewayClass | Gateways can be implemented by different controllers, often with different configurations. A Gateway object is associated with exactly one GatewayClass; the GatewayClass describes the gateway controller responsible for managing Gateways of this class +
GT | Gateway | A Gateway describes an instance of traffic handling infrastructure. It defines a network endpoint that can be used for processing traffic, i.e. filtering, balancing, splitting, etc. for backends such as a Service. For example, a Gateway may represent a cloud load balancer or an in-cluster proxy server that is configured to accept HTTP traffic. +
HTTP | HTTPRoute | The HTTPRoute kind specifies routing behavior of HTTP requests from a Gateway listener to backend network endpoints. For a Service backend, an implementation may represent the backend network endpoint as a Service IP or the backing EndpointSlices of the Service. An HTTPRoute represents configuration that is applied to the underlying Gateway implementation
****
TIP: Hover over each component in the diagram to learn what it does.

_Connectivity Link_ operates at the _Gateway API_ layer._Connectivity Link_ attaches its policies (AuthPolicy, TokenRateLimitPolicy) directly to these _Gateway API_ resources. This means security and governance are enforced **at the infrastructure level**, without modifying application code or the AI agent itself.
