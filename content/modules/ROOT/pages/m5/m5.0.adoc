== Lab 5 - Protect LLM consumption with Connectivity Link



In the previous modules, you built an integration platform that connects chat systems, ingests client data, and uses an AI agent to resolve support issues through natural language. But your organization runs more than one AI-powered tool.

Introducing the **writingAssistant**, which is a tool used by support agents to summarize customer conversations and draft professional email responses within the same Rocket.Chat interface. Under the hood, it relies on the **scribe agent** -- an LLM endpoint configured with enterprise-level quotas but no internal guardrails, no system prompt, and no memory. It's an open pipe to the model.

Some support agents have discovered this. Using adversarial prompts, they've started offloading massive personal tasks through the scribe -- summarizing long-form documents, generating personal content -- none of it related to customer support. The model works fine. The problem is **economic abuse**: unchecked token consumption is burning through the corporate API budget.

You might think traditional rate limiting solves this -- but it doesn't. Rate limiting counts _requests_, not _tokens_. Without token-level visibility, request-based limits can't stop the abuse.

In this lab, you'll secure the scribe agent's LLM endpoint by placing a **governance layer** in front of it using link:https://www.redhat.com/en/technologies/cloud-computing/connectivity-link[*_Red Hat Connectivity Link_*]. You'll implement two policies:

* **AuthPolicy** -- Control _who_ can access the endpoint using API key authentication and group-based authorization.
* **TokenRateLimitPolicy** -- Control _how much_ each user can consume by enforcing rate limits based on actual token usage.

{empty} +

=== Why Connectivity Link?

This is exactly the kind of problem _Red Hat Connectivity Link_ was designed for. Built on the link:https://kuadrant.io/[_Kuadrant_ project], it brings AI-aware governance to _Kubernetes_:

* **Token-aware rate limiting** -- limits are based on actual tokens consumed, not request counts. This is the only way to stop the kind of economic abuse the scribe agent enables.
* **Zero code changes** -- policies are applied at the infrastructure level. Your scribe agent and integration code stay untouched.
* **Kubernetes-native** -- policies are standard CRDs, applied with `oc apply`, stored in Git, reviewed in PRs. No separate management console or vendor control plane.
* **Composable** -- one policy extracts identity, another uses it for per-user token budgets. They layer on top of each other declaratively.

These capabilities come from how Connectivity Link builds on the _Kubernetes Gateway API_ standard. Before you start configuring policies, let's look at the Gateway API model.

{empty} +

=== The Gateway API Model

Connectivity Link layers on top of the _Kubernetes Gateway API_, the standard for managing traffic routing in _Kubernetes_. Understanding this model will make the exercises that follow intuitive.

Here are the building blocks:

* **GatewayClass + Gateway** -- owned by the platform team. Defines _how_ traffic enters the cluster. Already provisioned for you.
* **HTTPRoute** -- owned by the platform or development team. Defines _where_ your traffic goes (in this case, the LLM API).

This is where Connectivity Link comes in. Rather than embedding governance in your application, it attaches **policies** directly to these Gateway API resources. Authentication Policy and TokenRateLimit Policy layer onto the HTTPRoute -- enforced at the infrastructure level, invisible to your services:
