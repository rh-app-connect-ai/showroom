== Introduction

In the previous phases of this workshop, you connected disparate chat platforms into a unified communication hub, securely ingested and transformed client data, and introduced an AI agent that helps support agents resolve invoice issues through natural language -- all powered by Apache Camel, Kafka, and Model-as-a-Service (MaaS).

But providing internal teams with access to Large Language Models introduces new challenges. AI resource costs are tied to **token consumption** rather than simple API hits. Without proper governance, there is no way to prevent system abuse or control runaway expenses. Traditional rate limiting -- which simply counts requests -- is not enough.

In this final phase, you will place a **governance layer** in front of the LLM using **Red Hat Connectivity Link**. You will:

* **AuthPolicy** -- Control _who_ can access the LLM API using API key-based authentication and group-based authorization.
* **TokenRateLimitPolicy** -- Control _how much_ each user can consume by enforcing rate limits based on actual LLM token usage.

By the end of this module, the AI agent will be fully secured with tiered access control -- _free_ users get a limited token budget, while _gold_ users enjoy a higher allowance.

== Introducing Red Hat Connectivity Link

=== What is Connectivity Link?

Red Hat Connectivity Link is a Kubernetes-native solution for managing, securing, and controlling API traffic. Built on the **Kuadrant** project, it integrates directly with the **Gateway API** to enforce policies at the gateway level and HTTP Route Level -- before requests ever reach the LLM backend.

=== Gateway API Architecture

Red Hat Connectivity Link operates at the **Gateway API** layer -- a Kubernetes standard for managing ingress traffic. It uses a layered model:

[source]
----
GatewayClass          <-- Defines the controller (e.g., OpenShift Gateway Controller)
    |
    v
Gateway               <-- The actual listener (port, protocol, namespace scope)
    |
    v
HTTPRoute             <-- Routing rules (path matching, header modification, backend targets)
----

Connectivity Link attaches its policies (AuthPolicy, TokenRateLimitPolicy) directly to these Gateway API resources. This means security and governance are enforced **at the infrastructure level**, without modifying application code or the AI agent itself.

=== Understanding the User Secrets

Two API key secrets have been pre-created. Understanding their structure is important because both policies reference the labels and annotations on these secrets.

**Free-tier user:**
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: trlp-tutorial-api-key-free-user-1
  namespace: kuadrant-system
  labels:
    authorino.kuadrant.io/managed-by: authorino  # <1>
    app: my-llm                                   # <2>
  annotations:
    kuadrant.io/groups: free                       # <3>
    secret.kuadrant.io/user-id: {username}             # <4>
stringData:
  api_key: iamafreeuser
type: Opaque
----
<1> Marks this secret as managed by Authorino (the auth engine behind Connectivity Link).
<2> Label selector -- the AuthPolicy uses `app: my-llm` to discover valid API keys.
<3> Group membership -- determines which rate limit tier applies.
<4> Unique user ID -- used as a counter key for per-user rate limiting.

The **gold-tier** secret follows the same structure but with `kuadrant.io/groups: gold`,`api_key: iamagolduser`.

== Lab 1: Securing the API with Authentication

Now that you understand the architecture, it's time to lock down the LLM endpoint. You will create an **AuthPolicy** that enforces:

1. **API Key Authentication** -- Only requests with a valid Bearer token (matching a known secret) are allowed.
2. **Group-based Authorization** -- Only users belonging to the `free` or `gold` groups can access the API.
3. **Identity Extraction** -- The user's group and user ID are extracted from the secret's annotations and passed downstream for use by the TokenRateLimitPolicy.

=== Step 1: Review the AuthPolicy

Examine the AuthPolicy you are about to apply:

[source,yaml]
----
apiVersion: kuadrant.io/v1
kind: AuthPolicy
metadata:
  name: ai-proxy-auth
  namespace: {username}-ns                         # <1>
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route                           # <2>
  rules:
    authentication:
      api-key-users:
        apiKey:
          selector:
            matchLabels:
              app: my-llm                          # <3>
        credentials:
          authorizationHeader:
            prefix: Bearer                         # <4>
    response:
      success:
        filters:
          identity:
            json:
              properties:
                groups:
                  selector: auth.identity.metadata.annotations.kuadrant\.io/groups    # <5>
                userid:
                  selector: auth.identity.metadata.annotations.secret\.kuadrant\.io/user-id  # <6>
    authorization:
      allow-groups:
        opa:
          rego: |                                  # <7>
            groups := split(object.get(input.auth.identity.metadata.annotations, "kuadrant.io/groups", ""), ",")
            allow { groups[_] == "free" }
            allow { groups[_] == "gold" }
----
<1> The policy is created in your namespace (`{username}-ns`).
<2> Targets the `ai-proxy-route` HTTPRoute -- any request matching this route must pass authentication.
<3> Looks for Kubernetes Secrets labeled `app: my-llm` to find valid API keys.
<4> Expects the API key in the `Authorization` header with the `Bearer` prefix (e.g., `Authorization: Bearer iamafreeuser`).
<5> Extracts the `kuadrant.io/groups` annotation from the matched secret and exposes it as `groups` in the response filter.
<6> Extracts the `secret.kuadrant.io/user-id` annotation and exposes it as `userid`.
<7> OPA (Open Policy Agent) Rego rule that only allows users whose group is `free` or `gold`.

== Lab 2: Enforcing Token Budgets with Rate Limiting

Now that access is secured, you will add a **TokenRateLimitPolicy** to control how many LLM tokens each user can consume. This is the key differentiator for AI workloads -- unlike traditional rate limiting (which counts requests), token rate limiting counts the actual **tokens used** in LLM responses, giving you fine-grained cost control over AI consumption.

=== Step 1: Review the TokenRateLimitPolicy

Examine the policy you are about to apply:

[source,yaml]
----
apiVersion: kuadrant.io/v1alpha1
kind: TokenRateLimitPolicy
metadata:
  name: openai-token-limits
  namespace: {username}-ns                         # <1>
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route                           # <2>
  limits:
    free:                                          # <3>
      rates:
        - limit: 50
          window: 1m
      when:
        - predicate: request.path == "/v1/chat/completions"        # <4>
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "free") # <5>
      counters:
        - expression: auth.identity.userid                         # <6>
    gold:                                          # <7>
      rates:
        - limit: 200
          window: 1m
      when:
        - predicate: request.path == "/v1/chat/completions"
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "gold")
      counters:
        - expression: auth.identity.userid
----
<1> Created in your namespace (`{username}-ns`).
<2> Targets the same `ai-proxy-route` HTTPRoute as the AuthPolicy.
<3> **Free tier** -- limited to **50 tokens per minute**.
<4> Only applies to requests hitting the `/v1/chat/completions` path.
<5> Matches users whose `groups` identity property (set by the AuthPolicy response filter) contains `free`.
<6> Rate limit counter is **per user** -- each user gets their own token budget based on their `userid`.
<7> **Gold tier** -- limited to **200 tokens per minute**, same structure as free but with a higher allowance.

[IMPORTANT]
====
Notice how the TokenRateLimitPolicy depends on the AuthPolicy:

* The `auth.identity.groups` and `auth.identity.userid` fields referenced here are populated by the AuthPolicy's `response.success.filters` section.
* Without the AuthPolicy, the TokenRateLimitPolicy would have no identity context to evaluate.

This is a key pattern in Connectivity Link -- **policies compose together** through the Gateway API.
====

== Summary

Congratulations! You have completed the final phase of the workshop, adding a governance layer on top of the AI agent you built across the previous phases.

=== Key Takeaways

* **Connectivity Link policies attach to Gateway API resources** (HTTPRoutes, Gateways) -- no changes to the AI agent or application code required.
* **Policies compose together** -- the AuthPolicy extracts identity information that the TokenRateLimitPolicy uses for per-user, per-tier rate limiting.
* **Token-based rate limiting** is purpose-built for AI/LLM workloads, giving you cost control based on actual token consumption rather than simple request counts.
* **Everything is declarative and Kubernetes-native** -- policies are YAML resources managed with `kubectl`, making them easy to version, review, and automate.
* **Integration tools provide the foundation for AI** -- As you've seen across all four phases, robust connectivity tools like Apache Camel, Kafka, Service Interconnect, and Connectivity Link provide the essential, secure foundation needed to successfully weave AI into everyday business workflows.
