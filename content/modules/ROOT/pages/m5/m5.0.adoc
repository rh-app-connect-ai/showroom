== Lab 5 - Protect LLM consumption with Connectivity Link

== Introduction

In the previous phases of this workshop, you connected disparate chat platforms into a unified communication hub, securely ingested and transformed client data, and introduced an AI agent that helps support agents resolve invoice issues through natural language -- all powered by _Apache Camel_, _Kafka_, and Model-as-a-Service (MaaS).

But providing internal teams with access to Large Language Models introduces new challenges. AI resource costs are tied to **token consumption** rather than simple API hits. Without proper governance, there is no way to prevent system abuse or control runaway expenses. Traditional rate limiting -- which simply counts requests -- is not enough.

In this final phase, you will place a **governance layer** in front of the LLM using _Red Hat Connectivity Link_. You will:

* **AuthPolicy** -- Control _who_ can access the LLM API using API key-based authentication and group-based authorization.
* **TokenRateLimitPolicy** -- Control _how much_ each user can consume by enforcing rate limits based on actual LLM token usage.

By the end of this module, the AI agent will be fully secured with tiered access control -- _free_ users get a limited token budget, while _gold_ users enjoy a higher allowance.

== Introducing Red Hat Connectivity Link

=== What is Connectivity Link?

_Red Hat Connectivity Link_ is a _Kubernetes_-native solution for managing, securing, and controlling API traffic. Built on the _Kuadrant_ project, it integrates directly with the _Gateway API_ to enforce policies at the gateway level and HTTP Route Level -- before requests ever reach the LLM backend.

{empty} +

=== Gateway API Architecture

_Connectivity Link_ operates at the _Gateway API_ layer -- a _Kubernetes_ standard for managing ingress traffic. It uses a layered model:

[source]
----
GatewayClass          <-- Defines the controller (e.g., OpenShift Gateway Controller)
    |
    v
Gateway               <-- The actual listener (port, protocol, namespace scope)
    |
    v
HTTPRoute             <-- Routing rules (path matching, header modification, backend targets)
----

_Connectivity Link_ attaches its policies (AuthPolicy, TokenRateLimitPolicy) directly to these _Gateway API_ resources. This means security and governance are enforced **at the infrastructure level**, without modifying application code or the AI agent itself.

=== Understanding the User Secrets

Two API key secrets have been pre-created. Understanding their structure is important because both policies reference the labels and annotations on these secrets.

**Free-tier user:**

[.copypaste]
****
apiVersion: v1
kind: Secret
metadata:
  name: trlp-tutorial-api-key-free-{user-username}
  namespace: kuadrant-system
  labels:
    authorino.kuadrant.io/managed-by: authorino
    app: my-llm
  annotations:
    kuadrant.io/groups: free
    secret.kuadrant.io/user-id: {user-username}
stringData:
  api_key: iamafreeuser
type: Opaque
****

Key fields in this secret:

* `authorino.kuadrant.io/managed-by: authorino` -- Marks this secret as managed by _Authorino_ (the auth engine behind _Connectivity Link_).
* `app: my-llm` -- Label selector -- the AuthPolicy uses this to discover valid API keys.
* `kuadrant.io/groups: free` -- Group membership -- determines which rate limit tier applies.
* `secret.kuadrant.io/user-id: {user-username}` -- Unique user ID -- used as a counter key for per-user rate limiting.

The **gold-tier** secret follows the same structure but with `kuadrant.io/groups: gold`,`api_key: iamagolduser`.

== Lab 1: Securing the API with Authentication

Now that you understand the architecture, it's time to lock down the LLM endpoint. You will create an **AuthPolicy** that enforces:

1. **API Key Authentication** -- Only requests with a valid Bearer token (matching a known secret) are allowed.
2. **Group-based Authorization** -- Only users belonging to the `free` or `gold` groups can access the API.
3. **Identity Extraction** -- The user's group and user ID are extracted from the secret's annotations and passed downstream for use by the TokenRateLimitPolicy.

{empty} +

=== Step 1: Review the AuthPolicy

Examine the AuthPolicy you are about to apply. Copy the YAML below -- you will paste it in Step 2:

[.copypaste]
****
apiVersion: kuadrant.io/v1
kind: AuthPolicy
metadata:
  name: ai-proxy-auth
  namespace: {user-username}-devspaces
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route
  rules:
    authentication:
      api-key-users:
        apiKey:
          selector:
            matchLabels:
              app: my-llm
        credentials:
          authorizationHeader:
            prefix: Bearer
    response:
      success:
        filters:
          identity:
            json:
              properties:
                groups:
                  selector: auth.identity.metadata.annotations.kuadrant\.io/groups
                userid:
                  selector: auth.identity.metadata.annotations.secret\.kuadrant\.io/user-id
    authorization:
      allow-groups:
        opa:
          rego: |
            groups := split(object.get(input.auth.identity.metadata.annotations, "kuadrant.io/groups", ""), ",")
            allow { groups[_] == "free" }
            allow { groups[_] == "gold" }
****

Key fields in this policy:

* `namespace: {user-username}-ns` -- The policy is created in your namespace.
* `name: ai-proxy-route` -- Targets the `ai-proxy-route` HTTPRoute -- any request matching this route must pass authentication.
* `app: my-llm` -- Looks for _Kubernetes_ Secrets with this label to find valid API keys.
* `prefix: Bearer` -- Expects the API key in the `Authorization` header (e.g., `Authorization: Bearer iamafreeuser`).
* `kuadrant\.io/groups` -- Extracts the group annotation from the matched secret and exposes it as `groups` in the response filter.
* `secret\.kuadrant\.io/user-id` -- Extracts the user ID annotation and exposes it as `userid`.
* `rego` -- OPA (Open Policy Agent) rule that only allows users whose group is `free` or `gold`.

== Lab 2: Enforcing Token Budgets with Rate Limiting

Now that access is secured, you will add a **TokenRateLimitPolicy** to control how many LLM tokens each user can consume. This is the key differentiator for AI workloads -- unlike traditional rate limiting (which counts requests), token rate limiting counts the actual **tokens used** in LLM responses, giving you fine-grained cost control over AI consumption.

{empty} +

=== Step 1: Review the TokenRateLimitPolicy

Examine the policy you are about to apply. Copy the YAML below -- you will paste it in Step 2:

[.copypaste]
****
apiVersion: kuadrant.io/v1alpha1
kind: TokenRateLimitPolicy
metadata:
  name: openai-token-limits
  namespace: {user-username}-devspaces
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route
  limits:
    free:
      rates:
        - limit: 50
          window: 1m
      when:
        - predicate: request.path == "/v1/chat/completions"
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "free")
      counters:
        - expression: auth.identity.userid
    gold:
      rates:
        - limit: 200
          window: 1m
      when:
        - predicate: request.path == "/v1/chat/completions"
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "gold")
      counters:
        - expression: auth.identity.userid
****

Key fields in this policy:

* `namespace: {user-username}-ns` -- Created in your namespace.
* `name: ai-proxy-route` -- Targets the same HTTPRoute as the AuthPolicy.
* `free` -- **Free tier** -- limited to **50 tokens per minute**.
* `request.path == "/v1/chat/completions"` -- Only applies to requests hitting the chat completions path.
* `auth.identity.groups` -- Matches users whose `groups` identity property (set by the AuthPolicy response filter) contains `free` or `gold`.
* `auth.identity.userid` -- Rate limit counter is **per user** -- each user gets their own token budget.
* `gold` -- **Gold tier** -- limited to **200 tokens per minute**, same structure as free but with a higher allowance.

[IMPORTANT]
====
Notice how the TokenRateLimitPolicy depends on the AuthPolicy:

* The `auth.identity.groups` and `auth.identity.userid` fields referenced here are populated by the AuthPolicy's `response.success.filters` section.
* Without the AuthPolicy, the TokenRateLimitPolicy would have no identity context to evaluate.

This is a key pattern in _Connectivity Link_ -- **policies compose together** through the _Gateway API_.
====

== Summary

Congratulations! You have completed the final phase of the workshop, adding a governance layer on top of the AI agent you built across the previous phases.

=== Key Takeaways

* **_Connectivity Link_ policies attach to _Gateway API_ resources** (HTTPRoutes, Gateways) -- no changes to the AI agent or application code required.
* **Policies compose together** -- the AuthPolicy extracts identity information that the TokenRateLimitPolicy uses for per-user, per-tier rate limiting.
* **Token-based rate limiting** is purpose-built for AI/LLM workloads, giving you cost control based on actual token consumption rather than simple request counts.
* **Everything is declarative and _Kubernetes_-native** -- policies are YAML resources managed with `kubectl`, making them easy to version, review, and automate.
* **Integration tools provide the foundation for AI** -- As you've seen across all four phases, robust connectivity tools like _Apache Camel_, _Kafka_, Service Interconnect, and _Connectivity Link_ provide the essential, secure foundation needed to successfully weave AI into everyday business workflows.
