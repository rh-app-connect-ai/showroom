== Lab 5 - Protect LLM consumption with Connectivity Link

{banner-under-construction}

In the previous labs, you unified chat platforms, ingested client data, and introduced an AI agent that resolves invoice issues through natural language -- all powered by _Apache Camel_, _Kafka_, and LLMs.



But giving teams access to Large Language Models introduces a new challenge: **cost control**. AI costs are driven by token consumption, not simple API hits. Traditional rate limiting -- which just counts requests -- isn't enough to prevent abuse or runaway expenses.

In this lab, you'll place a **governance layer** in front of the LLM using _Red Hat Connectivity Link_ and implement two policies:

* **AuthPolicy** -- Control _who_ can access the LLM API using API key authentication and group-based authorization.
* **TokenRateLimitPolicy** -- Control _how much_ each user can consume by enforcing rate limits based on actual token usage.

By the end, the AI agent will have tiered access control -- _free_ users get a limited token budget, while _gold_ users get a higher allowance.

{empty} +

=== Introducing Red Hat Connectivity Link

_Red Hat Connectivity Link_ is a _Kubernetes_-native solution for managing, securing, and controlling API traffic. Built on the link:https://kuadrant.io/[_Kuadrant_ project], it integrates with the _Gateway API_ to enforce policies at the gateway and HTTP Route level -- before requests ever reach the LLM backend. It works in a single cluster or across multiple clusters.

{empty} +

=== Gateway API Architecture

The _Kubernetes Gateway API_ standardizes how traffic routing is managed within Kubernetes clusters. Created as a more flexible alternative to the Ingress API, it handles ingress, load balancing, and traffic routing using a layered model:

[.mermaid-hints-simple]
****
GC | GatewayClass | Defines the behavior and configuration of Gateways in a cluster. Specifies which controller implementation(istio, Openshift gateway controller ) will manage the Gateway. The GatewayClass also contains configurations like IP pools and resource limits. The infrastructure provider is responsible for defining and managing the GatewayClass   +
GT | Gateway | The entry point to your cluster (e.g. a cloud load balancer or ClusterIP Service) that directs traffic to the right destinations. +
HTTP | HTTPRoute | The routing rules for HTTP traffic -- it maps requests from the Gateway to your backend services(LLMs) based on URL paths, headers, or hosts.
****
TIP: Hover over each component in the diagram to learn what it does.

{empty} +

_Connectivity Link_ attaches its policies (AuthPolicy, TokenRateLimitPolicy) directly to these _Gateway API_ resources. Security and governance are enforced **at the infrastructure level** -- no changes to application code or the AI agent itself.
