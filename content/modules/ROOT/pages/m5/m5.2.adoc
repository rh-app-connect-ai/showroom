=== Auth Policy Powered by Connectivity Link
Now that you understand the concepts and the intended architecture, it's time to lock down the LLM endpoint. In this exercise you will create an **AuthPolicy** that is applied to the HTTPRoute(`ai-proxy-route`)  and enforces a key-based authentication scheme. This will ensure that only authorized users with a valid API key can access the LLM, and it will extract identity information that we can use for rate limiting in the next step.

NOTE: We are using simple API key authentication for this workshop to keep things straightforward and simple, but _Connectivity Link_ supports a wide range of authentication methods, including JWT, OAuth2 introspection, and more.

=== Understanding the User Secrets

Two API key secrets have already been *pre-created*. Understanding their structure is important because both policies reference the labels and annotations on these secrets.

**Free-tier user:**
[source,yaml,subs="attributes"]
----
apiVersion: v1
kind: Secret
metadata:
  name: trlp-tutorial-api-key-free-{username}
  namespace: kuadrant-system
  labels:
    authorino.kuadrant.io/managed-by: authorino
    app: my-llm
  annotations:
    kuadrant.io/groups: free
    secret.kuadrant.io/user-id: {username}
stringData:
  api_key: iamafree{username}
type: Opaque
----

Key fields in this secret:

* `authorino.kuadrant.io/managed-by: authorino` -- Marks this secret as managed by _Authorino_ (the auth engine behind _Connectivity Link_).
* `app: my-llm` -- Label selector -- the AuthPolicy uses this to discover valid API keys.
* `kuadrant.io/groups: free` -- Group membership -- determines which rate limit tier applies.
* `secret.kuadrant.io/user-id: {user-username}` -- Unique user ID -- used as a counter key for per-user rate limiting.

The **gold-tier** secret follows the same structure but with `kuadrant.io/groups: gold`,`api_key: iamagolduser`.

== Securing the LLM with Authentication

Now that you understand the architecture and the setup, it's time to lock down the LLM endpoint. You will create an **AuthPolicy** that is applied to the `ai-proxy-route` HTTPRoute and enforces:

1. **API Key Authentication** -- Only requests with a valid Bearer token (matching a known secret) are allowed.
2. **Group-based Authorization** -- Only users belonging to the `free` or `gold` groups can access the API.
3. **Identity Extraction** -- The user's group and user ID are extracted from the secret's annotations and passed downstream for use by the TokenRateLimitPolicy that we will create in the next section.

{empty} +

=== Step 1: Review and Apply the AuthPolicy

Examine the AuthPolicy and *apply* it by pasting the following command in your terminal:

[.copypaste]
****
oc apply -f - <<EOF
apiVersion: kuadrant.io/v1
kind: AuthPolicy
metadata:
  name: ai-proxy-auth
  namespace: {user-username}-devspaces
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route
  rules:
    authentication:
      api-key-users:
        apiKey:
          selector:
            matchLabels:
              app: my-llm
        credentials:
          authorizationHeader:
            prefix: Bearer
    response:
      success:
        filters:
          identity:
            json:
              properties:
                groups:
                  selector: auth.identity.metadata.annotations.kuadrant\.io/groups
                userid:
                  selector: auth.identity.metadata.annotations.secret\.kuadrant\.io/user-id
    authorization:
      allow-groups:
        opa:
          rego: |
            groups := split(object.get(input.auth.identity.metadata.annotations, "kuadrant.io/groups", ""), ",")
            allow { groups[pass:[_]] == "free" }
            allow { groups[pass:[_]] == "gold" }
EOF
****

Key fields in this policy:

* `namespace: {user-username}-devspaces` -- The policy is created in your namespace.
* `name: ai-proxy-route` -- Targets the `ai-proxy-route` HTTPRoute -- any request matching this route must pass authentication.
* `app: my-llm` -- Looks for _Kubernetes_ Secrets with this label to find valid API keys.
* `prefix: Bearer` -- Expects the API key in the `Authorization` header (e.g., `Authorization: Bearer iamafreeuser`).
* `kuadrant\.io/groups` -- Extracts the group annotation from the matched secret and exposes it as `groups` in the response filter.
* `secret\.kuadrant\.io/user-id` -- Extracts the user ID annotation and exposes it as `userid`.
* `rego` -- OPA (Open Policy Agent) rule that only allows users whose group is `free` or `gold`.
