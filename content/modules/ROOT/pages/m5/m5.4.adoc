== Token Rate Limiting

With authentication in place, the next layer of control is token consumption. A **TokenRateLimitPolicy** tracks the actual *tokens used* in LLM responses, rather than simply counting requests like traditional rate limiting.

Rate limiting policies are a powerful way to combine AI capabilities with cost and resource control. In this exercise, you'll see how the Customer Profile Tool uses token budgets to decide which tier of analysis to produce.

Every LLM request consumes compute power, and richer analysis consumes more of it. As AI adoption grows, managing cost and infrastructure capacity becomes critical. Rate limiting policies help organisations control token usage across their services.

In this exercise, the Customer Profile Tool goes beyond simple rate limiting with a graceful degradation strategy. It first attempts a *Gold* analysis, and when the Gold budget is consumed, falls back to *Silver* to keep producing useful profiles at a lower cost. Only when both are exhausted is the request rejected:

=== Apply and Verify the TokenRateLimitPolicy

The TokenRateLimitPolicy is a gateway-level resource. It defines token budgets per tier and lets the gateway enforce them independently of the running application.

. Open a new terminal tab in _DevSpaces_ by clicking the kbd:[+] icon (keep the previous tab with _Camel_ running so you can test later).
+
[#terminal-tab]

[.overlay]
****
[.target]
terminal-tab

[.arrows]
{x: 38, y: 67, rot: -45, size: 3}
****
+
{empty} +
+
NOTE: Because the policy operates at the infrastructure level, you can apply it while the agent is running, with no restart required.
+
{empty} +

. Examine the token budgets for each tier, then apply the policy to your namespace:
+
--
The policy defines two token budgets that reflect the cost of each tier:

* *Gold* (`pro`) — richer analysis, higher compute cost. A stricter token allowance with a slower renewal window.
* *Silver* (`lite`) — lighter analysis, lower compute cost. A more generous token allowance that renews faster.
--
+
{blank}
+
Review the TokenRateLimitPolicy below and *apply it by pasting* the following yaml in your new terminal tab:
+
{blank}
+
[.copypaste]
****
oc apply -f - <<EOF
apiVersion: kuadrant.io/v1alpha1
kind: TokenRateLimitPolicy
metadata:
  name: openai-token-limits
  namespace: {user-username}-devspaces
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route
  limits:
    pro:
      rates:
        - limit: 50
          window: 4m
      when:
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "pro")
      counters:
        - expression: auth.identity.userid
    lite:
      rates:
        - limit: 500
          window: 2m
      when:
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "lite")
      counters:
        - expression: auth.identity.userid
EOF
****
+
Notice how the YAML matches the tier budgets described above: `pro` with a tighter limit and longer window, `lite` with a more generous allowance that renews faster.
+
[IMPORTANT]
====
The TokenRateLimitPolicy depends on the AuthPolicy. The `auth.identity.groups` and `auth.identity.userid` fields referenced here are populated by the AuthPolicy's `response.success.filters` section. Without the AuthPolicy, the TokenRateLimitPolicy would have no identity context to evaluate.
====
+
