== Enforcing Token Budgets with Rate Limiting

{banner-under-construction}

Now that access is secured, you will add a **TokenRateLimitPolicy** to control how many LLM tokens each user can consume. This is the key differentiator for AI workloads -- unlike traditional rate limiting (which counts requests), token rate limiting counts the actual **tokens used** in LLM responses, giving you fine-grained cost control over AI consumption.

{empty} +

=== Step 1: Review and Apply the AuthPolicy
. Open a new terminal tab in your dev space (keep the previous tab with the chat running so you can test later).
+
image::m5/devspace_terminal_new.png[Dev Spaces terminal - placeholder,width=80%]

. Examine the TokenRateLimitPolicy and *apply* it by pasting the following command in your terminal:
+
[.copypaste]
****
oc apply -f - <<EOF
apiVersion: kuadrant.io/v1alpha1
kind: TokenRateLimitPolicy
metadata:
  name: openai-token-limits
  namespace: {user-username}-devspaces
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route
  limits:
    free:
      rates:
        - limit: 50
          window: 1m
      when:
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "free")
      counters:
        - expression: auth.identity.userid
    gold:
      rates:
        - limit: 200
          window: 1m
      when:
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "gold")
      counters:
        - expression: auth.identity.userid
EOF
****

Key fields in this policy:

* `namespace: {user-username}-devspaces` -- Created in your namespace.
* `name: ai-proxy-route` -- Targets the same HTTPRoute as the AuthPolicy.
* `free` -- **Free tier** -- limited to **50 tokens per minute**.
* `request.path == "/v1/chat/completions"` -- Only applies to requests hitting the chat completions path.
* `auth.identity.groups` -- Matches users whose `groups` identity property (set by the AuthPolicy response filter) contains `free` or `gold`.
* `auth.identity.userid` -- Rate limit counter is **per user** -- each user gets their own token budget.
* `gold` -- **Gold tier** -- limited to **200 tokens per minute**, same structure as free but with a higher allowance.

[IMPORTANT]
====
Notice how the TokenRateLimitPolicy depends on the AuthPolicy:

* The `auth.identity.groups` and `auth.identity.userid` fields referenced here are populated by the AuthPolicy's `response.success.filters` section.
* Without the AuthPolicy, the TokenRateLimitPolicy would have no identity context to evaluate.
====

=== Step 3: Verify the TokenRateLimitPolicy

. From the devspaces terminal, run the following command to check that the AuthPolicy was created successfully:
+
[.copypaste]
****
oc describe tokenratelimitpolicy -n {user-username}-devspaces | grep -iE "Name:|target|free:|gold:|limit:|window:|type:|status:"
****

. You should see output confirming the AuthPolicy exists, is targeting the correct HTTPRoute, and is in an enforced state, similar to:
----
Name:         openai-token-limits
    Free:
        Limit:   50
        Window:  1m
    Gold:
        Limit:   200
        Window:  1m
  Target Ref:
    Name:   ai-proxy-route
Status:
    Status:                True
    Type:                  Accepted
    Status:                True
    Type:                  Enforced
----



[.verification]
****
Does the  TokenRateLimitPolicy show status **Enforced**?

[.success]
The token rate limit policy is active. Proceed to test token limits.

[.fail]
Check that you pasted the correct YAML and that the namespace is set to `{user-username}-devspaces`.
****

=== Step 4: Test Token Rate Limiting via the Chat Interface

Use the chat interface to test the rate limiting behavior.

{empty} +

==== Test 1: Free-tier token limits

. Send a message that asks the LLM to generate a response that consumes around 50 tokens. Note: While token counts can vary, the below prompt is designed to elicit a response of approximately 50 tokens.
+
[.copypaste]
****
Explain Kubernetes in 50 words
****
+
{empty} +

. Send the same message again as soon as you get the response back.
+
[.copypaste]
****
Explain Kubernetes in 50 words
****
+
{empty} +

. Your second request should be **rate limited** (429 Too Many Requests), as the user would have crossing the limit of consuming **50 tokens within a minute**,.
+
image::m5/trlp_free_rate_limited.png[Free-tier rate limited response - placeholder,width=80%]
{empty} +

. Wait for the 1-minute window to reset, then try again -- the request should succeed.


{empty} +

{empty} +

[NOTE]
====
Rate limits are tracked **per user** using the `userid` counter from the AuthPolicy. Each user gets their own independent token budget -- one user hitting their limit does not affect another user's quota.
====
