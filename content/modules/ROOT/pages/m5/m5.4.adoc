== Enforcing Token Budgets with Rate Limiting



Now that access is secured, you will add a **TokenRateLimitPolicy** to control how many LLM tokens each user can consume when using the writing assistant. This is the key differentiator for AI workloads -- unlike traditional rate limiting (which counts requests), token rate limiting counts the actual **tokens used** in LLM responses, giving you fine-grained cost control over AI consumption.

Now let's assume that one of the support agents has realized that the calls to the scribe agent are not protected by any rate limits, and they start using it to summarize long documents and generate personal content, burning through the team's API budget. Let's see how the TokenRateLimitPolicy can help prevent this kind of economic abuse by enforcing token budgets for different user groups.

=== Review and Apply the TokenRateLimitPolicy
. Open a new terminal tab in your dev space by clicking the "+" icon (keep the previous tab with the chat running so you can test later).
+
image::m5/devspace_terminal_new.png[Dev Spaces terminal - placeholder,width=80%]
+
{empty} +

. Examine the TokenRateLimitPolicy and *apply it by pasting* the following yaml in your terminal:
+
[.copypaste]
****
oc apply -f - <<EOF
apiVersion: kuadrant.io/v1alpha1
kind: TokenRateLimitPolicy
metadata:
  name: openai-token-limits
  namespace: {user-username}-devspaces
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: ai-proxy-route
  limits:
    lite:
      rates:
        - limit: 50
          window: 1m
      when:
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "lite")
      counters:
        - expression: auth.identity.userid
    pro:
      rates:
        - limit: 500
          window: 1m
      when:
        - predicate: |
            auth.identity.groups.split(",").exists(g, g == "pro")
      counters:
        - expression: auth.identity.userid
EOF
****

Key fields in this policy:

* `namespace: {user-username}-devspaces` -- Created in your namespace.
* `name: ai-proxy-route` -- Targets the same HTTPRoute as the AuthPolicy.
* `lite` -- **lite tier** -- limited to **50 tokens per minute**.
* `auth.identity.groups` -- Matches users whose `groups` identity property (set by the AuthPolicy response filter) contains `lite` or `pro`.
* `auth.identity.userid` -- Rate limit counter is **per user** -- each user gets their own token budget.
* `pro` -- **pro tier** -- limited to **500 tokens per minute**, same structure as lite but with a higher allowance.

[IMPORTANT]
====
Notice how the TokenRateLimitPolicy depends on the AuthPolicy:

* The `auth.identity.groups` and `auth.identity.userid` fields referenced here are populated by the AuthPolicy's `response.success.filters` section.
* Without the AuthPolicy, the TokenRateLimitPolicy would have no identity context to evaluate.
====

The agent masks their personal request in a prompt that looks work-related and invokes the scribe agent to generate the response, which then calls the protected LLM agent. The TokenRateLimitPolicy should kick in and prevent excessive token consumption.

====
