== Enabling Database Connectivity

Before implementing the data flow, you need a connection to the backend database. Let's see what happens when you try to reach it.

=== Bridging namespaces with Service Interconnect

{rhsi} solves this by creating encrypted application-level connections between namespaces — without requiring VPNs, firewall rules, or exposing services to the public network.

Once configured, a local service is created in your namespace that transparently forwards traffic to the remote database:

{blank}

=== Under the hood

Service Interconnect uses a set of resources to establish and manage the connection. Here's how the previous diagram breaks down:

{blank}

[.mermaid]
----
flowchart LR
    subgraph yours [" "]
        Client[DB Client]
        subgraph S2 ["Site(2)"]
            Listener((Listener))
            Grant[AccessGrant]
        end
    end

    Client --> Listener

    S2 -.-|"🔒 secure link"| S1

    subgraph shared [" "]
        subgraph S1 ["Site(1)"]
            Connector((Connector))
            Token[AccessToken]
        end
        SVC((postgres))
        SVC --- DB[(PostgreSQL)]
    end

    Connector --> SVC
    Grant ~~~ Token

    style S2 fill:#e8eaf6,stroke:#7986cb

    style Grant fill:#fff,stroke:#7986cb

{empty} +

Each side runs a *Site* — the foundation that enables the Skupper network. The connection between them is established through an *AccessGrant* and *AccessToken* pair. Traffic flows from the *Listener* in your namespace to the *Connector* on the database side.

NOTE: The database lives in a secure, air-tight environment — no inbound ports are exposed. To keep it that way, the link is initiated *from* the database side *outward* to your site. This is why you create an AccessGrant on your side: it gives the database site permission to connect to you and establish a bidirectional communication link. Its environment never opens an entrypoint that could be exploited.

{empty} +

=== What you will do

The server side has been partially configured by your platform team:

* A *Site* in the `shared-database` namespace
* A *Connector* that bridges the Skupper network to the PostgreSQL service on port `5432`

Your job is to complete the client side:

. Create a *Site* in your namespace with `linkAccess` enabled
. Generate an *AccessGrant* so the database namespace can connect to you
. Create an *AccessToken* in the database namespace to establish the link
. Create a *Listener* to expose the remote database as a local service

=== Step 1: Create a Site in Your Namespace

Create a Site with `linkAccess` enabled. This makes your namespace a *listening site* that can accept incoming connections from the database.

{blank}

[.copypaste]
****
cat <<EOF | oc apply -f -
apiVersion: skupper.io/v2alpha1
kind: Site
metadata:
  name: {username}-site
  namespace: {username}-devspaces
spec:
  linkAccess: default
EOF
****

Verify the Site is ready:

=== Step 2: Create an AccessGrant

Before jumping in, remember that the link is initiated *from* Site(1) (database) *to* Site(2) (yours). Here's how the grant and token work together to make that happen:

{blank}

[.mermaid]
----
sequenceDiagram
    actor A as Admin<br/>(You)
    participant S2 as Site(2)
    participant S1 as Site(1)

    A-->>S2: Create AccessGrant

    Note over S2: AccessGrant

    A-->>S1: Create AccessToken

    Note over S1: AccessToken

    S1->>S2: Redeem token
    S2->>S1: Transfer link details
    S1->>S2: Create link

    Note over S2,S1: 🔒 secure link established
----

You create an AccessGrant on your Site(2) and an AccessToken on Site(1). The database site then redeems the token to establish a secure link back to you.

{blank}

Now, create the AccessGrant:

{blank}

[.copypaste]
****
cat <<EOF | oc apply -f -
apiVersion: skupper.io/v2alpha1
kind: AccessGrant
metadata:
  name: grant-{username}
  namespace: {username}-devspaces
spec:
  redemptionsAllowed: 1
  expirationWindow: 168h
EOF
****

Wait for the AccessGrant to be ready and retrieve its details:

{blank}

[.copypaste]
****
oc get accessgrant grant-{username} -n {username}-devspaces
****

Once ready, the grant's `.status` will contain the `url`, `code`, and `ca` fields needed for the next step.

=== Step 3: Create an AccessToken in the Database Namespace

Now create an AccessToken in the `shared-database` namespace to establish the link from the database to your namespace. The token must be named `token-{username}`.

Extract the grant details and create the AccessToken:

{blank}

[.copypaste]
****
URL=$(oc get accessgrant grant-{username} -n {username}-devspaces -o jsonpath='{.status.url}')
CODE=$(oc get accessgrant grant-{username} -n {username}-devspaces -o jsonpath='{.status.code}')
CA=$(oc get accessgrant grant-{username} -n {username}-devspaces -o jsonpath='{.status.ca}' | sed 's/^/    /')

cat <<EOF | oc apply -f -
apiVersion: skupper.io/v2alpha1
kind: AccessToken
metadata:
  name: token-{username}
  namespace: shared-database
spec:
  url: "${URL}"
  code: "${CODE}"
  ca: |
${CA}
EOF
****

IMPORTANT: The AccessToken name *must* follow the pattern `token-<your-username>`. Any other name will be rejected.

After a few moments, a *Link* resource is automatically created in `shared-database`. Verify:

{blank}

[.copypaste]
****
oc get link -n shared-database
****

Wait until the link status shows `Ready`.

=== Step 4: Create a Listener to Expose the Database Locally

The Listener creates a local service in your namespace that forwards traffic to the remote PostgreSQL database through the encrypted link.

{blank}

[.copypaste]
****
cat <<EOF | oc apply -f -
apiVersion: skupper.io/v2alpha1
kind: Listener
metadata:
  name: postgres-listener
  namespace: {username}-devspaces
spec:
  routingKey: postgres
  host: postgres
  port: 5432
EOF
****

IMPORTANT: The `routingKey` must match the one configured on the Connector in the `shared-database` namespace. The platform team set it to `postgres`.

Verify the Listener is ready:

{blank}

=== Cleanup

To tear down the Service Interconnect resources you created, delete them in reverse order:

{blank}

[.copypaste]
****
oc delete accesstoken token-{username} -n shared-database
oc delete listener postgres-listener -n {username}-devspaces
oc delete accessgrant grant-{username} -n {username}-devspaces
oc delete site {username}-site -n {username}-devspaces
****

This removes the link, listener, grant, and site. The server-side resources in `shared-database` (Site, Connector) are not affected.
