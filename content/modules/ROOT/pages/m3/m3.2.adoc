== Kafka to Rocket.Chat

{banner-under-construction}

To connect _Matrix_ and _Rocket.Chat_ through _Kafka_, you need two data flows:

{empty} +

[.mermaid]
----
flowchart LR
    M([Matrix]) -->|"flow 1"| K([Kafka])
    K -->|"flow 2"| RC([Rocket.Chat])

    style M fill:#e8f5e9,stroke:#66bb6a
    style K fill:#fff3e0,stroke:#fb8c00
    style RC fill:#e8f5e9,stroke:#66bb6a

    linkStyle 0 stroke:#ccc,stroke-width:1px
    linkStyle 1 stroke:#d32f2f,stroke-width:3px
----

[.text-center]
✏️ You are here: *flow 2 — Kafka to Rocket.Chat*

{empty} +

You'll start with the second flow — the listener side. By setting it up first, the receiving end will be ready. When you wire up the _Matrix_ to _Kafka_ flow in the next section, you'll immediately see messages flowing end-to-end.

{empty} +

The flow is already prepared — you just need to fill in three areas:

. Define _Kafka_ as the source system
. Define a filter to prevent loopback events
. Define the data mapping to transform _Kafka_ events for _Rocket.Chat_

{blank}

The picture below highlights the focus areas:

image::m3/flow-k2r-todo.jpg[width=80%]

{empty} +

[NOTE]
--
*What is a loopback event?*

All systems publish to and listen from a single _Kafka_ topic. This means a message sent from _Rocket.Chat_ ends up in the same topic this flow is consuming — and could circle right back:

[.mermaid]
----
flowchart LR
    RC(["Rocket.Chat"]) -- "message" --> K(["Kafka topic"]) -- "❌ same event" --> RC

    style K fill:#fff3e0,stroke:#fb8c00
    style RC fill:#e8f5e9,stroke:#66bb6a

    linkStyle 0 stroke:#66bb6a,stroke-width:2px
    linkStyle 1 stroke:#d32f2f,stroke-width:2px,stroke-dasharray:5
----

The filter you'll define prevents this by checking the `source` field in each event and blocking those that originate from _Rocket.Chat_.
--

=== Define Kafka as the source system

The first step of the flow is where you define _Kafka_ as the source from where events arrive.

NOTE: Your _OpenShift_ namespace already contains an _Streams for Apache Kafka_ cluster (_Kafka_) ready for you to use.

1. Hover the pass:[<span style="color: #B0380C;">&#x2757; To do: Kafka</span>] step and click the floating {btn-replace} button.
1. Find and select the _Kafka_ component.
1. Select the `All` configuration tab
1. Configure the following 2 properties:
+
`Topic`:
+
{blank}
+
[.copypaste]
****
myroom
****
+
`Brokers`:
+
{blank}
+
[.copypaste]
****
my-cluster-kafka-bootstrap:9092
****
+

=== Define the filter

Events arriving from _Kafka_ can originate from any source system. Without a filter, a message originally sent from _Rocket.Chat_ would flow through _Kafka_ and be pushed back to _Rocket.Chat_ — creating a loop:

  * _Rocket.Chat -> Kafka -> Rocket.Chat_ ⟵ loopback!

{blank}

The filter prevents this by checking where the event came from.

1. Click the bar pass:[<code style="background-color: #B0380C; color: white;">&nbsp;To do: Filter&nbsp;</code>]
+
You should see the `JQ` expression language selected by default.
+
1. Copy and paste the following expression:
+
{blank}
+
[.copypaste]
****
pass:[.source | test("rocketchat")]
****
+
NOTE: The expression picks the `source` field from the _Kafka_ event and checks if it originates from _Rocket.Chat_. Events from _Rocket.Chat_ are filtered out to prevent loopback.
+
{empty} +

=== Define the data transformation

The data arriving from _Kafka_ uses a common schema. You need to map it to the format _Rocket.Chat_ expects.

{empty} +

. Include in your Camel route a Kaoto data transformation step
+
--
From Kaoto:

. Click the kbd:[+] button inside the pass:[<code style="background-color: #4494E5; color: white;">&nbsp;To do: Mapping&nbsp;</code>] step
. Type `mapper` in the filter to find the _Kaoto DataMapper_
. Click the _Kaoto DataMapper_ tile
. Click the mapper to open the configuration panel
. Click the pass:a[<span {btn-stl-blue}>{map-wrench}  Configure</span>] button
--
+
{empty} +

. Next, define the inputs and outputs
+
--
From the DataMapper editor:

. Define the source:
.. On **Parameters** click the button kbd:[+] to add new parameter
.. Enter `common` and click the checkmark button kbd:[{map-tick}]
.. Click kbd:[{map-att}] (Attach schema), next click
pass:a[<span {btn-stl-blue-outline}>{map-upload} Upload schema file(s)</span>] and then pick at the top of your screen: `support/schemas/*common.json*`.
.. Click pass:a[<span {btn-stl-blue}>Attach</span>] to complete the action.
+
{blank}
+
. Define the target
.. On the right hand side of the DataMapper screen: +
Click kbd:[{map-att}] (Attach schema), next click
pass:a[<span {btn-stl-blue-outline}>{map-upload} Upload schema file(s)</span>] and then pick at the top of your screen: `support/schemas/*rocketchat-out.json*`.
.. Click pass:a[<span {btn-stl-blue}>Attach</span>] to complete the action.

. Define the data mapping rules
+
Copy and Paste (in the *Target*) the values below to define the given field entries:
+
... `string[@key = channel]`:
+
[.copypaste]
****
[.path]
/roomid

[.default]
room11

"REPLACE"
****
+
{blank}

... `string[@key = text]`:

[.copypaste]
****

****

[TIP]

.What does this command do?
[%collapsible]
====
* `camel cmd send` — sends a message to a running Camel process
* `k2r` — the name of the target process
* `--body '{...}'` — the JSON payload simulating a Kafka event with `user`, `channel`, `source`, and `text` fields
